# Runbook: panther (Pixel 7) — gs201 kernel + ReSukiSU + SUSFS + ZeroMount

Инструкция для Claude Code. Цель — собрать загружаемое ядро для **Pixel 7 (panther, SoC gs201)** на базе
**WildKernels gs201**, привив в него **ZeroMount (VFS) + SUSFS**, совместимые с **ReSukiSU**, и упаковать
AnyKernel3. Прошивку и тест на устройстве делает человек, не ты.

---

## 0. Контекст и цель

- Устройство: Pixel 7, codename `panther`, SoC `gs201`.
- Ветка ядра: **GKI android14-6.1** (panther остаётся на ней и под Android 16 — это нормально).
- Стоковый саблевел на устройстве: **6.1.145**, Android 16 userspace.
- Root: **ReSukiSU** (форк KernelSU, менеджер совместим с KSU/MKSU/RKSU/SukiSU).
- Скрытие: **SUSFS** + **ZeroMount** в VFS-режиме (драйвер `/dev/zeromount`, перехват `getname()`).
- База сборки: `WildKernels` gs201 (исходное дерево tensynos/gs201, которое УЖЕ загружается на panther с KSU+SUSFS).
- Источник патчей: `Enginex0/Super-Builders`, каталог `android14-6.1/`.

### Принцип №1 — минимальный диф к рабочей базе
WildKernels gs201 уже грузится на panther. Меняем как можно меньше. Всё device-специфичное
(модули gs201, упаковка, способ прошивки) НЕ трогаем — используем существующий workflow дерева как есть,
только добавляем патчи и флаги конфига.

### Принцип №2 — собирать из gs201-дерева, не из generic GKI
Image и модули gs201 должны собираться из одного дерева, иначе KMI разъедется и panther не загрузится.
Это причина, по которой мы НЕ используем generic-GKI пайплайн Super-Builders напрямую — берём только его патчи.

### Принцип №3 — ZeroMount жёстко связан с SUSFS из `50_`
Патч `60_zeromount` написан против конкретной версии SUSFS из патча `50_enhanced_susfs` (общие биты inode-флагов,
`AS_FLAGS` collision guards, supercall-проводка). Нельзя смешивать `60_` с ПРОИЗВОЛЬНОЙ версией SUSFS.
Это главный риск интеграции — см. Фазу 2.

### Принцип №4 — про саблевел 145
«Оставить 145» = собранное ядро должно грузиться на стоке 6.1.145. Дерево WildKernels может стоять на ≥145
(напр. 6.1.16x) — это допустимо: GKI грузит равный-или-старший саблевел при той же KMI-генерации.
НЕ нужно насильно даунгрейдить дерево до ровно 145. uname при необходимости спуфится через SUSFS.
Сверь только KMI-генерацию (`android14-XX`) дерева со стоком.

## Правила работы (соблюдать строго)

1. **НИКОГДА не флашить, не запускать `fastboot flash`/`fastboot boot`.** Ты только собираешь и пакуешь.
2. На каждом **CHECKPOINT** — остановись, выведи указанный артефакт/лог и **жди подтверждения человека**.
   Не переходи к следующей фазе сам.
3. Все изменения — в форке, отдельная ветка `zeromount-panther`. Коммить осмысленными шагами.
4. Если патч ложится «с фаззом» (fuzz) или есть `.rej` — это НЕ «успех». Останавливайся и репортуй.
5. Ничего не «додумывай» про железо panther. Сомнения — в отчёт человеку.
6. Не правь упаковку/скрипты прошивки WildKernels без явного указания.

## 1. Recon (CHECKPOINT 1)

Не меняй ничего. Только собери факты.

```bash
# 1.1 База WildKernels gs201 — найти точный репозиторий и подтвердить таргет
#     (вероятно WildKernels/Sultan_KernelSU_SUSFS или WildKernels gs201 kernel/CI-репо)
gh repo view WildKernels/Sultan_KernelSU_SUSFS
# подтвердить: targets gs201/panther? ветка android14-6.1? как собирает (workflow + build entrypoint)?

# 1.2 Какой KSU-форк уже в дереве и как подключён (submodule / setup.sh / in-tree)?
# 1.3 Есть ли в дереве SUSFS и какая версия? Найти define версии:
grep -rni "SUSFS_VERSION\|susfs.*v1\.\|SUS_SU" --include=*.c --include=*.h | head
# 1.4 Где defconfig для panther/gs201? (arch/arm64/configs, gki_defconfig, device fragment)
# 1.5 Какой workflow собирает gs201 и какие у него dispatch-инпуты?
ls -la .github/workflows
```

```bash
# 1.6 Источник патчей
gh repo view Enginex0/Super-Builders
# подтвердить наличие и точные имена:
#   android14-6.1/ReSukiSU/patches/50_enhanced_susfs-android14-6.1.patch
#   android14-6.1/ReSukiSU/patches/60_zeromount-android14-6.1.patch
#   android14-6.1/ReSukiSU/patches/70_ksu_safety-*-6.1.patch
#   android14-6.1/.../defconfig.fragment
#   build-helpers/
# Найти версию SUSFS, против которой написан 50_ (внутри патча — define/строка версии).
```

**STOP — CHECKPOINT 1. Вывести человеку отчёт:**
- точный репо+ветка базы WildKernels gs201 и команда/воркфлоу сборки;
- какой KSU-форк сейчас в дереве и способ подключения;
- **версия SUSFS в дереве WildKernels** vs **версия SUSFS в `50_` Super-Builders** (это решает стратегию);
- путь(и) defconfig для panther;
- KMI-генерация дерева (`android14-XX`) и его текущий саблевел.

Жди решения по стратегии SUSFS (Фаза 2) от человека.

## 2. Стратегия SUSFS + KSU (CHECKPOINT 2)

Развилка по результатам recon. Решение принимает человек, ты предлагаешь.

**По SUSFS:**
- **Вариант A (мало конфликтов):** версия SUSFS в WildKernels == версии под `50_`.
  → НЕ применяем `50_`. Используем SUSFS дерева, добавляем только `70_` + `60_`.
- **Вариант B (правильнее, но дороже):** версии расходятся.
  → Снимаем/отключаем SUSFS-интеграцию WildKernels и кладём матч-сет `50_` + `70_` + `60_`.
  Так гарантируется совместимость inode-флагов с ZeroMount.

**По KSU:** цель — **ReSukiSU**, потому что `70_ksu_safety` написан под него.
- Если в дереве другой форк (напр. Wild_KSU) — заменить на ReSukiSU:
  ```bash
  # удалить текущую KSU-интеграцию по правилам дерева, затем:
  curl -LSs "https://raw.githubusercontent.com/ReSukiSU/ReSukiSU/main/kernel/setup.sh" | bash -s <branch>
  # точную команду/branch взять из доков ReSukiSU
  ```
- KSU должен лежать ДО применения `70_` (патч правит код внутри `kernel/` форка).

**STOP — CHECKPOINT 2. Вывести человеку:**
- выбранную стратегию SUSFS (A или B) с обоснованием;
- план по KSU (оставляем/заменяем на ReSukiSU) и точную setup-команду;
- список файлов, которые затронет интеграция (предварительная оценка конфликтов).

Жди «go».

## 3. Наложение патчей (CHECKPOINT 3)

Ветка `zeromount-panther`. Порядок строго как в Super-Builders: **`50_` → `70_` → `60_`**
(в варианте A — `70_` → `60_`). Проверяй каждый патч ПЕРЕД применением.

```bash
git checkout -b zeromount-panther

# для каждого патча: сухой прогон, затем применение с тремя путями отката
git apply --check -p1 <patch>            # должно молчать
git apply --reject  -p1 <patch>          # создаст *.rej при конфликте
# или, если дерево предпочитает: patch -p1 --dry-run < <patch>

# после каждого:
git status --porcelain
find . -name "*.rej"                      # ДОЛЖНО быть пусто
```

Если есть `.rej` или фазз — НЕ коммить «как есть». Резолвь конфликт руками: открой `.rej`, пойми,
куда лёг бы хунк в gs201-дереве (там out-of-tree-код в namei/readdir/proc/namespace может смещать контекст),
внеси правку, удали `.rej`. Затем `git apply --check` чистого состояния.

Применить `build-helpers/` если они нужны под текущий саблевел (адаптация API между саблевелами).

**STOP — CHECKPOINT 3. Вывести человеку:**
- по каждому патчу: применился чисто / с фаззом / были `.rej`;
- содержимое всех `.rej`, если были, и как ты их разрулил (diff твоих ручных правок);
- `git diff --stat` итоговой ветки.

Жди ревью диффа. **Это самый важный чекпоинт для контроля смысла правок.**

## 4. defconfig + флаги (CHECKPOINT 4)

Влить `defconfig.fragment` из Super-Builders в defconfig panther/gs201 (НЕ перетирать, а слить).
Критично включить:
- SUSFS-конфиги (как в фрагменте);
- **CONFIG ZeroMount** — точный символ берётся ИЗ `defconfig.fragment`, не угадывай.
  Без него драйвер не соберётся и VFS-режим молча выпадет в OverlayFS-фолбэк;
- `CONFIG_KSU=y` (ReSukiSU), при желании `CONFIG_KPM=y`;
- проверь, что хук ReSukiSU включён корректно для GKI2 (tracepoint syscall redirect или susfs inline hook),
  и не конфликтует с manual hook.

```bash
# показать только относящиеся к интеграции строки после слияния:
grep -nE "SUSFS|ZERO|ZEROMOUNT|KSU|KPM|KPROBES|TRACEPOINT" <defconfig_path>
```

**STOP — CHECKPOINT 4. Вывести человеку** результат `grep` выше (финальные значимые CONFIG-строки)
и подтверждение, что символ ZeroMount реально присутствует и `=y`. Жди «go».

## 5. Сборка через Actions (CHECKPOINT 5)

Использовать **существующий** workflow WildKernels gs201 (он умеет в device-модули и упаковку panther).
Только добавить шаг применения патчей/ветку, если workflow того требует. Не писать сборку с нуля.

```bash
gh workflow run <gs201_workflow>.yml --ref zeromount-panther <inputs...>
gh run watch
# при падении — вытащить хвост лога именно с первой ошибкой:
gh run view --log-failed
```

Типичные грабли, на которые смотреть в логах: несведённые символы SUSFS/ZeroMount (значит SUSFS-версии
разъехались — вернуться к Фазе 2, вариант B), `BUILD_BUG_ON` по `AS_FLAGS` (конфликт битов inode-флагов),
отсутствие toolchain/Kleaf-таргета (использовать тот, что в workflow дерева).

**STOP — CHECKPOINT 5. Вывести человеку:**
- статус сборки (success/fail);
- при ошибке — блок лога от первой ошибки (не весь лог);
- при успехе — список артефактов и имя итогового `AnyKernel3*.zip`, что в нём лежит (Image + модули gs201?).

Жди «go» на выдачу артефакта.

## 6. Выдача артефакта (твоя финальная задача)

При зелёной сборке — приложить ссылку на артефакт/релиз и краткий мэппинг:
какой саблевел, какая KMI-генерация, какой KSU-форк, включён ли ZeroMount VFS.
**На этом твоя часть заканчивается. Дальше — человек.**

---

## 7. Проверка на устройстве (делает человек, не Claude Code)

1. Бэкап: сохранить стоковые `boot` / `init_boot` / `vbmeta` (и `dtbo`, если используется).
2. **Сначала `fastboot boot` собранного boot.img** — НЕ `flash`. Если бутлуп — просто ребут в сток.
3. После загрузки проверить:
   - `ls -l /dev/zeromount` — узел есть;
   - в WebUI ZeroMount: **capability = VFS** (а не OverlayFS/MagicMount фолбэк);
   - радио/Wi-Fi/сеть/датчики/термал — живые (главный риск Pixel);
   - `uname -r` спуфится как сток (если включено);
   - mountinfo чистый для не-root (KSU-записи не светятся).
4. Только если всё ок несколько ребутов подряд — флашить в раздел.

---

## 8. Откат

- Бутлуп после `fastboot boot`: ребут — устройство грузится со стока, ничего не записано.
- После записи: `fastboot flash` стоковых `boot`/`init_boot`/`vbmeta` из бэкапа.

- **CP3:** статус по каждому патчу, все `.rej` и твои ручные правки, `git diff --stat`.
- **CP4:** `grep` финальных CONFIG-строк + подтверждение символа ZeroMount.
- **CP5:** статус сборки; при ошибке — лог с первой ошибки; при успехе — состав zip.
